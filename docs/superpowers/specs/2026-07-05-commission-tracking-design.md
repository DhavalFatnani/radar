# Commission Tracking — Design Spec

**Date:** 2026-07-05
**Phase:** 5, Slice 4
**Status:** Design locked, ready for implementation plan
**Source of truth:** `Phase0_Platform_Specification.md` §4.7 / §7.6 (projects + commission + leak-defense), `AGENTS.md` non-negotiables, existing `projects` schema skeleton.

---

## 1. Overview

When a lead is **won**, it becomes a *project* — a deal the operator brokered between a buyer (lead) and a vendor. The operator earns a **commission** on that deal. This slice makes commission a first-class, tracked entity on the lead-detail page: the operator records the commission **terms** when the deal is won, the commission becomes **due** when the deal is **delivered**, and the operator **records payment manually** when the vendor pays. The slice also tracks the *leak-defense* audit trail — what contact details were disclosed, when introductions were made, and any dispute — so the operator has dated proof of origination if a vendor tries to bypass the platform.

This is the **full feature** (per the locked scope decision): one-time **and** recurring commissions, active due-cycle surfacing, missed-payment flags, and the three leak-defense logs — all in this slice.

## 2. Locked directional decisions

These four product decisions were made by the operator and are **not** open for reinterpretation during implementation:

| # | Decision | Value |
|---|----------|-------|
| 1 | **Basis** | **Support both, per record.** Each commission chooses *percentage-of-deal-value* **or** *flat amount*. |
| 2 | **Lifecycle** | **Terms at `won` → due at `delivered` → paid manually.** Terms are set when the lead is marked won; the amount becomes collectible when the deal reaches delivered; the operator marks it paid when the money arrives. |
| 3 | **Currency** | **INR only.** Single currency; all money stored as integer **minor units (paise)**. No multi-currency, no FX. |
| 4 | **Scope** | **Full feature now.** One-time + recurring, with active due-cycle surfacing, missed-payment flags, and disclosure / introduction / dispute logs. Nothing deferred to a later slice. |

## 3. Goals / Non-goals

**Goals**
- Record commission terms (type, basis, deal value or flat amount, rate, cadence) against a won lead.
- Generate the expected payment **cycles** (one for one-time; a recurring series for recurring) and make them **due** when the deal is delivered.
- Let the operator mark a cycle **paid**, **missed**, or **waived**, and add the next recurring cycle as it comes around.
- Surface, on read, which cycles are **due now** and which are **overdue** (the "reminder" and "missed-payment flag" behaviour) without any background scheduler.
- Maintain three append-only leak-defense logs (disclosure, introduction, dispute) with dated entries.
- Present all of this in a `commission-panel.tsx` on the lead-detail page, mirroring the existing `outreach-panel.tsx` composition and a11y patterns.

**Non-goals**
- No background jobs / cron / email reminders. "Reminders" are computed-on-read badges in the panel. (A future slice may add scheduled notifications; out of scope here.)
- No multi-currency, no tax/GST computation, no invoice-document generation.
- No changes to how a lead advances through pipeline stages — the existing `advanceLeadStageAction` / `setLeadStage` stay as they are. Commission activation is driven by its own commission action, keyed off the lead's current stage, so the pipeline module stays decoupled.
- No commission analytics / reporting rollups across projects (single-project panel only).

## 5. Data model

The `projects` table already exists as an all-jsonb skeleton (`project_id`, `lead_id`, `vendor_id`, plus six speculative jsonb columns) and has **never been written to** — the table is empty. We redefine it into a coherent set. Because the table is empty, the migration is pure DDL with no data backfill.

### 5.1 `projects` table (post-migration)

| column | type | notes |
|--------|------|-------|
| `project_id` | `uuid` PK default random | unchanged |
| `lead_id` | `uuid` NOT NULL → `leads.lead_id` | unchanged |
| `vendor_id` | `uuid` NOT NULL → `vendor_profiles.vendor_id` | unchanged |
| `commission_status` | `commission_status` enum NOT NULL default `'pending'` | **NEW typed column** — project-level lifecycle |
| `commission_terms` | `jsonb` (nullable) | `CommissionTerms` (Zod-validated) |
| `commission_cycles` | `jsonb` NOT NULL default `'{"cycles":[]}'` | `{ cycles: CommissionCycle[] }` (Zod-validated) — renamed from skeleton `commission_due` |
| `disclosure_log` | `jsonb` NOT NULL default `'[]'` | `DisclosureEntry[]` |
| `introduction_log` | `jsonb` NOT NULL default `'[]'` | `IntroductionEntry[]` |
| `dispute_log` | `jsonb` NOT NULL default `'[]'` | `DisputeEntry[]` — renamed from skeleton `dispute_record` for array semantics |

Skeleton columns `commission_due`, `recurring_tracking`, and `dispute_record` are removed/renamed by the migration (the table is empty, so drop+add is safe). One row per won lead (a project). Unique constraint on `lead_id` (a lead maps to at most one project).

**Money is never a typed column** — all amounts live inside the jsonb payloads as plain integers (paise). Crore-scale deals are safe: ₹100 crore = 1e11 paise, well under `Number.MAX_SAFE_INTEGER` (9.007e15) and valid JSON.

### 5.2 `commission_status` enum (project-level lifecycle)

`commission_status = ['pending', 'active', 'closed', 'disputed', 'void']`

| value | meaning |
|-------|---------|
| `pending` | lead won, terms set, deal **not yet delivered** — no cycle is collectible |
| `active` | deal delivered, one or more cycles due / being collected |
| `closed` | every expected cycle is paid or waived — commission fully settled |
| `disputed` | a dispute is open (set via the dispute action) |
| `void` | commission cancelled (e.g. lead reverted / deal fell through) |

### 5.3 Zod types & constants (in `schema.ts`, pure)

```ts
// mirrors the DB enum, single source for the union + label map
export const COMMISSION_STATUSES = ['pending','active','closed','disputed','void'] as const;
export type CommissionStatus = (typeof COMMISSION_STATUSES)[number];

export const COMMISSION_TYPES = ['one_time','recurring'] as const;
export type CommissionType = (typeof COMMISSION_TYPES)[number];

export const COMMISSION_BASES = ['percentage','flat'] as const;
export type CommissionBasis = (typeof COMMISSION_BASES)[number];

export const RECURRING_CADENCES = ['monthly','quarterly','annual'] as const;
export type RecurringCadence = (typeof RECURRING_CADENCES)[number];

export const CYCLE_STATUSES = ['scheduled','due','paid','missed','waived'] as const;
export type CommissionCycleStatus = (typeof CYCLE_STATUSES)[number];
```

**`commissionTermsSchema`** — a discriminated / super-refined object enforcing the basis+type invariants:

- `type: CommissionType`, `basis: CommissionBasis`.
- basis `percentage` ⇒ `dealValueInr: int ≥ 0` **and** `rateBps: int 1..10000` (basis points; 1000 = 10%). `amountInr` absent.
- basis `flat` ⇒ `amountInr: int ≥ 0`. `dealValueInr` / `rateBps` absent.
- type `recurring` ⇒ `cadence: RecurringCadence` required. type `one_time` ⇒ `cadence` absent.

**`CommissionCycle`**:

```ts
{
  seq: number;                 // 1-based sequence
  dueDate: string;             // ISO date "YYYY-MM-DD"
  amountInr: number;           // paise, computed at generation from terms
  status: CommissionCycleStatus;
  paidAt?: string | null;      // ISO datetime, set when marked paid
  paidAmountInr?: number | null;
}
```

`commissionCyclesSchema = z.object({ cycles: z.array(commissionCycleSchema) })`.

**Leak-defense entries** (append-only arrays, each entry carries a dated proof timestamp — AGENTS.md non-negotiable #1):

```ts
DisclosureEntry   = { at: string; contactField: string; disclosedTo: string; note?: string }
IntroductionEntry = { at: string; channel: string; note?: string }
DisputeEntry      = { openedAt: string; reason: string; status: 'open' | 'resolved'; resolvedAt?: string | null; resolution?: string | null }
```

### 5.4 Pure functions (in `schema.ts`, unit-tested, no DB, no clock)

Time is **injected**, never read from the ambient clock, so functions are deterministic and testable (mirrors the codebase's date-passing convention).

- `computeCycleAmountInr(terms): number` — `percentage`: `Math.floor(dealValueInr * rateBps / 10000)`; `flat`: `amountInr`.
- `buildInitialCycles(terms, startDate: string): CommissionCycle[]` — `one_time`: exactly one cycle, `seq:1`, `dueDate:startDate`, amount from `computeCycleAmountInr`, `status:'scheduled'`. `recurring`: first cycle `seq:1` at `startDate` (subsequent cycles are added on demand via the add-next-cycle action, not pre-generated — YAGNI on an unbounded horizon).
- `nextCycleDueDate(cadence, fromDate): string` — adds 1 month / 3 months / 12 months to `fromDate`.
- `isCycleOverdue(cycle, today: string): boolean` — `cycle.status === 'due' && cycle.dueDate < today`.
- `deriveCommissionStatus(current, cycles): CommissionStatus` — never overrides `disputed`/`void`; otherwise: all cycles `paid`/`waived` and at least one exists ⇒ `closed`; any cycle `due`/`missed` ⇒ `active`; else keep `current`.
- `activateCycles(cycles): CommissionCycle[]` — flips every `scheduled` cycle to `due` (used at delivered).
- `formatInr(paise): string` — display helper, e.g. `1234500 → "₹12,345.00"` (Indian grouping). Pure; used by the panel.

## 6. Business rules & lifecycle

1. **Set terms (at `won`).** The panel shows a "Set commission terms" form once the lead's `pipelineStage` is one of `won` / `delivered` / `paid` (an explicit set — **not** an ordinal comparison, because the enum places `lost` after `won`) and no project exists. Submitting creates the `projects` row: `commission_terms` stored, `commission_status = 'pending'`, `commission_cycles = buildInitialCycles(terms, today)` all in `scheduled` status. Terms may be edited while status is `pending`.
2. **Activate (at `delivered`).** When the lead's stage is `delivered` (or `paid`), the panel enables an "Activate commission (deal delivered)" control. It flips `scheduled` cycles to `due` (`activateCycles`) and sets `commission_status = 'active'`. Guarded: rejected unless the lead stage is actually `delivered`/`paid`.
3. **Record payment.** The operator marks a due cycle **paid** (records `paidAt = now`, `paidAmountInr`). After the write, `commission_status` is recomputed via `deriveCommissionStatus`; if all cycles are settled it becomes `closed`.
4. **Recurring continuation.** For a recurring project, an "Add next cycle" control appends a new `due` cycle at `nextCycleDueDate(cadence, lastDueDate)` with the per-cycle amount, keeping status `active`.
5. **Missed / waived.** A due cycle past its `dueDate` is surfaced as **overdue** (`isCycleOverdue`, computed on read) — this is the missed-payment flag. The operator may explicitly mark a cycle **missed** (record-keeping) or **waived** (excuse it; counts as settled for `closed` derivation).
6. **Disputes.** "Open dispute" appends a `DisputeEntry{status:'open'}` and sets `commission_status = 'disputed'`. "Resolve dispute" sets the latest entry `resolved` (+ `resolvedAt`, `resolution`) and recomputes status back via `deriveCommissionStatus`.
7. **Disclosure / introduction logs.** Append-only add-entry actions; purely additive audit trail, never mutate/delete existing entries.

**Reminders without a scheduler:** the panel computes, for `today`, each cycle's due/overdue state and renders badges ("Due", "Overdue N days"). No emails, no cron. `today` is passed from the server component into the pure helpers.

## 7. Data layer (`src/lib/commission/data.ts`)

All functions take `db: DB` (type-only import of the `DB` type, as in `pipeline/data.ts`), guard UUIDs, use parameterized Drizzle (`eq(...)`), return `{ ok: true; ... } | { ok: false; error: string }` for mutations, and never throw on domain errors (DB errors propagate).

```ts
export async function getCommissionForLead(db: DB, leadId: string): Promise<CommissionRecord | null>
export async function createCommissionTerms(db: DB, leadId: string, vendorId: string, terms: CommissionTerms, today: string): Promise<Result>
export async function updateCommissionTerms(db: DB, leadId: string, terms: CommissionTerms, today: string): Promise<Result>  // only when status 'pending'
export async function activateCommission(db: DB, leadId: string): Promise<Result>       // scheduled → due, status active
export async function markCyclePaid(db: DB, leadId: string, seq: number, paidAmountInr: number, now: string): Promise<Result>
export async function markCycleMissed(db: DB, leadId: string, seq: number): Promise<Result>
export async function waiveCycle(db: DB, leadId: string, seq: number): Promise<Result>
export async function addNextCycle(db: DB, leadId: string): Promise<Result>            // recurring only
export async function appendDisclosure(db: DB, leadId: string, entry: DisclosureEntry): Promise<Result>
export async function appendIntroduction(db: DB, leadId: string, entry: IntroductionEntry): Promise<Result>
export async function openDispute(db: DB, leadId: string, reason: string, at: string): Promise<Result>
export async function resolveDispute(db: DB, leadId: string, resolution: string, at: string): Promise<Result>
```

`CommissionRecord` = the row's typed shape (status + parsed terms + cycles + logs). Jsonb is read back through the Zod schemas; malformed payloads degrade to safe defaults (empty cycles / logs), matching `getLeadDetail`'s tolerant-parse convention. Cycle mutations read-modify-write the `commission_cycles` jsonb (single-operator app; no concurrency concern).

## 8. Actions (`src/app/(app)/leads/[id]/actions.ts`, extend)

Each is `"use server"`, returns `Promise<{ ok: boolean; error?: string }>`, and follows the guard ladder **auth-first**:

```
auth() → not signed in? → load lead → lead missing? → stage guard → input validation (Zod) → data-layer call → revalidatePath(`/leads/${leadId}`) on success
```

Actions:
- `setCommissionTermsAction(leadId, termsInput)` — stage must be `won`+; validates `commissionTermsSchema`; create-or-update.
- `activateCommissionAction(leadId)` — stage must be `delivered`/`paid`.
- `markCyclePaidAction(leadId, seq, paidAmountInr)`
- `markCycleMissedAction(leadId, seq)` / `waiveCycleAction(leadId, seq)`
- `addNextCycleAction(leadId)`
- `appendDisclosureAction` / `appendIntroductionAction` / `openDisputeAction(leadId, reason)` / `resolveDisputeAction(leadId, resolution)`

Timestamps (`now`/`today`) are produced in the action (server) and passed into the pure/data layer, so the data + pure layers stay clock-free. Errors returned to the client are safe constant strings — never DB internals or stack traces (global security constraint).

## 9. UI (`commission-panel.tsx` + page wiring)

`CommissionPanel` is a client component mirroring `OutreachPanel`: a `run(action)` `useTransition` helper, `router.refresh()` on success, `role="alert"` error region, real `<button>`s, `role="group"` + `aria-label`, inline two-step confirms (no native `confirm()`) for irreversible marks (paid / missed / waived / open-dispute). Mobile-first (375px), semantic HTML, keyboard-navigable, focus states.

Rendered on the lead-detail page **after** `<ContactBlockView>`, only when `lead.pipelineStage` is one of the explicit set `{won, delivered, paid}` (`lost` is excluded even though it sorts after `won` in the enum). Props are plain values fetched server-side.

Panel regions:
- **No project yet** (stage in `{won, delivered, paid}`, no row): "Set commission terms" form — type (one-time/recurring), basis (percentage/flat), conditional fields (deal value + rate, or flat amount), cadence (if recurring). Client-side + server-side Zod validation.
- **Project exists:** status badge; terms summary (formatted ₹); "Activate" control if `pending` and stage delivered; cycles table (seq, due date, amount ₹, status, overdue badge) with per-cycle mark-paid / mark-missed / waive controls; "Add next cycle" for recurring active projects.
- **Leak-defense:** three collapsible log sections (disclosure / introduction / dispute) each showing dated entries + an add-entry form; open/resolve dispute controls.

`page.tsx` fetches the commission via `getCommissionForLead(db, id)` alongside `getLeadDetail`, computes `today` server-side, and passes both to `<CommissionPanel>`.

## 10. Migration

- Add `commission_status` enum + column to `projects`.
- Rename/replace jsonb columns: drop `commission_due`, `recurring_tracking`, `dispute_record`; add `commission_cycles`, `dispute_log` (with NOT NULL defaults); keep `commission_terms`, `disclosure_log`, `introduction_log` (add NOT NULL defaults where appropriate).
- Add a UNIQUE constraint on `projects.lead_id`.
- Generated via `npm run db:generate` (drizzle-kit) into `src/db/migrations/00NN_<slug>.sql`; applied with `npm run db:migrate`. Integration tests pick it up automatically through `migrateTestDb()` (runs the real migrations against the Neon test branch). The table is empty in every environment, so no backfill.

## 12. Global constraints (carried into the plan verbatim)

- **INR minor units (paise)** everywhere; integer math; no floats for money; `formatInr` for display only.
- Money lives in **jsonb** (no bigint columns); values ≤ `Number.MAX_SAFE_INTEGER`.
- **Data-module split:** `schema.ts` pure (no `@/db`, no `server-only`, no `@/ai` value imports); `data.ts` server-only by import graph.
- **Injected `db`** first arg in every data function; **parameterized Drizzle** (`eq`), never string SQL; UUID guards on all ids.
- **Auth-first** guard ladder in every action; safe constant error strings only (no stack traces / DB internals to client).
- **Clock injected** into pure + data layers (timestamps produced in actions).
- **Dated proof** on every leak-defense entry (non-negotiable #1); leak logs append-only.
- **Layer separation:** panel imports only `schema.ts`; `data.ts` sole `@/db` toucher; `src/ai/` untouched.
- **a11y / mobile-first:** 375px-first, semantic HTML, keyboard-navigable, focus states, `role=alert` errors, inline confirms (no native `confirm()`).
- **TDD**, tests next to (or mirroring dir of) the code, ≥80% new-code coverage, non-vacuous tests.
