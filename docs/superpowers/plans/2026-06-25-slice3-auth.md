# Slice 3 — Single-Operator Access — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Gate the app behind a minimal, secure single-operator login (NextAuth v5 credentials, env-stored bcrypt hash) so unauthenticated requests cannot reach protected routes.

**Architecture:** NextAuth v5 (Auth.js) with a Credentials provider, using the **split-config** pattern: an Edge-safe `authConfig` (route-protection logic, no bcrypt) consumed by both `middleware.ts` and the Node `NextAuth` instance, plus a Node-only `verifyOperator` (bcrypt) added as the Credentials `authorize`. JWT sessions (no DB session table). The operator's email + bcrypt password hash live in env (no users table — multi-user stays parked per spec).

**Tech Stack:** next-auth@beta (v5 / Auth.js) · bcryptjs · zod (v3) · Next.js 15 App Router · Vitest.

**Source spec:** `Prompt_Playbook.md` Part 3 → Slice 3; `docs/superpowers/specs/2026-06-25-phase1-stack-design.md` (auth = NextAuth v5 credentials, single operator); operator-approved decision: **env-based bcrypt credentials** (no DB operators table).

## Global Constraints

Apply to every task.

- **One operator, no sign-up / roles / multi-user** (§1.3 parked SaaS list). Credentials are env-based.
- **Security (from `~/.claude/CLAUDE.md`):** secrets only via env, never in the repo; passwords stored as **bcrypt hashes**, never plaintext; never return stack traces / internal errors to the client; the login form must be keyboard-navigable with focus states and semantic HTML.
- **Auth route prefix is `/api/auth/*`** (Auth.js owns it) — this is the one allowed exception to the project's `/api/v1/*` convention. App API routes stay under `/api/v1/*`.
- **Edge/Node split:** `middleware.ts` runs on the Edge runtime — it must NOT import bcrypt or any Node-only code. Only `authConfig` (pure logic) is shared with the middleware; the Credentials provider + bcrypt live in the Node `src/lib/auth/index.ts`.
- **Session strategy: JWT** (the Credentials provider does not support database sessions).
- **Auth env is validated in the auth module** (`src/lib/auth/env.ts`), NOT in the central `src/lib/env.ts` — this keeps the DB layer and its tests decoupled from auth env.
- TypeScript strict; zod v3. Tests in `tests/unit/`.
- **Staging discipline:** stage only each task's files explicitly — never `git add -A` (a separate `mockups/` worktree, design specs, and `.superpowers/` scratch must stay out).

## Design Decisions (flagged for operator review)

1. **Env-based credentials** (operator-approved): `OPERATOR_EMAIL` + `OPERATOR_PASSWORD_HASH` (bcrypt) + `AUTH_SECRET`. No users/operators table.
2. **`verifyOperator(input, operator)` is a pure function** taking the operator config explicitly (not reading env) → fully unit-testable with no env setup. The NextAuth `authorize` callback injects the env values.
3. **Route protection via the `authorized` callback** in `authConfig` + a middleware matcher that runs on everything except Next internals, static assets, `/api/auth/*`, and `/api/v1/health`. Public paths: `/` and `/login`. Everything else requires a session.
4. **Protected area = the `(app)` route group.** This slice adds a placeholder `(app)/dashboard`; Slice 4 fills in the real nav sections (which are auto-protected by the same matcher).
5. **Login rate-limiting is deferred to Phase 6 (hardening)** — flagged, not built now (the `~/.claude/CLAUDE.md` "rate-limit public endpoints" rule applies to the eventual deployment; out of scope for a single-operator Phase-1 login).
6. **Auth env is required only at app runtime/build**, not for tests — the unit tests import `verifyOperator` (pure) and `authConfig` (no env), so `npm test` needs no auth env. `next build` does (it builds the auth route + middleware), so `.env.local` gets the values in Task 4.

- [ ] **Step 4: Implement `src/lib/auth/verify-operator.ts`**

```ts
import bcrypt from "bcryptjs";

export interface Operator {
  email: string;
  passwordHash: string;
}

export interface OperatorUser {
  id: string;
  email: string;
}

// Pure credential check for the single operator. Returns the user on an exact
// (case-insensitive email) match with a verified bcrypt password, else null.
// Takes the operator config explicitly so it is testable without env.
export async function verifyOperator(
  input: { email?: unknown; password?: unknown },
  operator: Operator,
): Promise<OperatorUser | null> {
  const email = typeof input.email === "string" ? input.email.trim().toLowerCase() : "";
  const password = typeof input.password === "string" ? input.password : "";
  if (!email || !password) return null;

  const operatorEmail = operator.email.trim().toLowerCase();
  if (email !== operatorEmail) return null;

  const ok = await bcrypt.compare(password, operator.passwordHash);
  if (!ok) return null;

  return { id: "operator", email: operatorEmail };
}
```

- [ ] **Step 6: Create `src/lib/auth/env.ts`**

```ts
import { z } from "zod";

// Auth env is validated HERE (not in src/lib/env.ts) so the DB layer and its
// tests stay decoupled from auth config. Evaluated only when the Node NextAuth
// instance (./index.ts) is imported — app runtime/build, never the unit tests.
const authEnvSchema = z.object({
  AUTH_SECRET: z.string().min(1),
  OPERATOR_EMAIL: z.string().email(),
  OPERATOR_PASSWORD_HASH: z.string().min(1),
});

export type AuthEnv = z.infer<typeof authEnvSchema>;

export function parseAuthEnv(source: NodeJS.ProcessEnv | Record<string, unknown>): AuthEnv {
  const result = authEnvSchema.safeParse(source);
  if (!result.success) {
    const issues = result.error.issues
      .map((issue) => `${issue.path.join(".")}: ${issue.message}`)
      .join("; ");
    throw new Error(`Invalid auth environment variables: ${issues}`);
  }
  return result.data;
}

export const authEnv = parseAuthEnv(process.env);
```

- [ ] **Step 3: Implement `src/lib/auth/config.ts` (Edge-safe)**

```ts
import type { NextAuthConfig } from "next-auth";

// Public paths reachable without a session. Everything else requires auth.
const PUBLIC_PATHS = new Set(["/", "/login"]);

// Edge-safe config: pure logic only — NO bcrypt, NO Node-only imports. Shared
// by middleware.ts (Edge) and extended by the Node instance in ./index.ts.
export const authConfig = {
  trustHost: true,
  session: { strategy: "jwt" },
  pages: { signIn: "/login" },
  providers: [], // the Credentials provider (Node + bcrypt) is added in ./index.ts
  callbacks: {
    authorized({ auth, request }) {
      const { pathname } = request.nextUrl;
      if (PUBLIC_PATHS.has(pathname) || pathname.startsWith("/api/auth")) return true;
      return !!auth?.user;
    },
  },
} satisfies NextAuthConfig;
```

```ts
"use server";

import { AuthError } from "next-auth";
import { signIn } from "@/lib/auth";

// Returns an error message string on failure, or never returns on success
// (signIn throws a NEXT_REDIRECT which must propagate). Never leaks internals.
export async function authenticate(
  _prevState: string | undefined,
  formData: FormData,
): Promise<string | undefined> {
  try {
    await signIn("credentials", {
      email: formData.get("email"),
      password: formData.get("password"),
      redirectTo: "/dashboard",
    });
  } catch (error) {
    if (error instanceof AuthError) {
      return "Invalid email or password.";
    }
    throw error; // re-throw the redirect (and anything unexpected)
  }
  return undefined;
}
```
